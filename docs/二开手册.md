# 医疗预约随访系统 · 二次开发手册

> 适用版本：v1.0.0 ｜ 技术栈：Fastify 4 + better-sqlite3 + Vue 3 + Pinia + Element Plus
> 本手册面向需要在现有系统上做二次开发（新增模块/字段/接口/页面）的工程师，覆盖架构解读、设计模式、扩展流程、测试规范与已知问题。

---

## 1. 系统概述

医疗病例随访排班系统，围绕「病例—随访—排班」三条主线，提供患者档案管理、随访记录跟踪、排班与冲突检测能力，并配套首页概览看板。

- **后端**：Node.js + Fastify 4，使用 better-sqlite3（WAL 模式）作为数据库，分层架构（Routes → Controllers → Services → Repositories）。
- **前端**：Vue 3（Composition API）+ Vite + Pinia + Element Plus + Axios，采用 composables 复用逻辑。
- **数据**：三张核心表 `patients` / `followups` / `schedules`，通过外键关联，首次启动自动写入种子数据。

## 5. 后端核心设计

### 5.1 分层职责

| 层 | 文件 | 职责 | 约定 |
|----|------|------|------|
| Routes | `routes/*.js` | 路由声明 | 纯函数 `(fastify) => {}`，只做方法/路径→控制器绑定，不写业务 |
| Controllers | `controllers/*Controller.js` | HTTP 适配 | 解析 `request.query/params/body`，调 service，用 `success/successPage/created` 包装返回 |
| Services | `services/*Service.js` | 业务规则 | 校验、默认值、关联校验、缓存失效（`_invalidateCache`），抛 `AppError` |
| Repositories | `repositories/*Repository.js` | 数据访问 | 继承 BaseRepository，组装 SQL 条件；跨表 JOIN 写在仓库内 |
| Base | [BaseRepository.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/repositories/BaseRepository.js) | 通用 CRUD | 提供 `findById/findOne/findAll/findWithKeyword/findPaginated/count/exists/create/update/patch/delete/raw*` |

### 5.2 工厂模式（懒加载单例）

三个工厂 [RepositoryFactory](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/repositories/index.js) / [ServiceFactory](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/services/index.js) / [ControllerFactory](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/controllers/index.js) 用 getter + 内部 `_xxx` 缓存实现懒加载单例，并在 [server.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/server.js#L26-L35) 通过 `fastify.decorate` 注入到实例，路由内以 `fastify.controllers.patient` 访问。

**二开新增模块时**：在对应工厂追加一个 getter 即可，无需改动装配代码。

### 5.3 统一响应封装

[utils/response.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/utils/response.js) 定义：

- `success(data, message, extra)` → `{ code:0, data, message, ...extra }`
- `successPage(list, total, page, pageSize)` → `{ code:0, data:{list,total,page,pageSize} }`
- `created(data, message)` → `{ code:0, data, message, _statusCode:201 }`
- `fail(message, code, data)` → `{ code, data, message }`
- `AppError` + `notFound` / `badRequest` / `conflict`：业务异常，携带 `statusCode/code/data`

业务码（[constants/index.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/constants/index.js#L45-L50)）：`SUCCESS=0`、`ERROR=-1`、`UNAUTHORIZED=-2`、`FORBIDDEN=-3`。

> 设计意图：由 [responseHandler](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/middleware/responseHandler.js) 插件的 `preSerialization` 剥离 `_statusCode` 并设置真实 HTTP 码，由 `setErrorHandler` 捕获 `AppError` 返回 `{code,message,data}`。**该意图当前未生效，原因见 [§13](#13-已知问题与修复建议)。**

### 5.4 缓存机制

[utils/cache.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/utils/cache.js) 是基于 `Map` 的命名空间缓存：

- 键形如 `namespace:key`，TTL 默认 5 分钟，上限 200 条，LRU 淘汰。
- `get/set/delete/invalidate(namespace)/clear/wrap(namespace,key,fn,ttl)`。
- Service 层用 `cache.wrap` 读、用 `_invalidateCache()` 写后失效。

**跨模块失效约定**：写随访会同时失效 `followups` 与 `patients` 命名空间（见 [FollowupService._invalidateCache](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/services/FollowupService.js#L11-L14)）；排班同理。二开新增模块时，若读取了其他模块数据，记得在写操作后失效对应命名空间。

短 TTL 用法：今日待办随访缓存 60 秒（[FollowupService.listTodayPending](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/services/FollowupService.js#L35-L40)）。

### 5.5 查询构建器

[utils/queryBuilder.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/utils/queryBuilder.js)：

- `buildPagination(page, pageSize)`：自动夹取 `1..maxPageSize(200)`，返回 `{limit,offset,page,pageSize}`。
- `buildWhereClause(conditions)`：`conditions` 为 `{field,operator,value,modifier}` 数组，支持 `= LIKE IN IS IS_NOT >= <= > < CUSTOM`；空值自动跳过。
- `buildKeywordCondition(keyword, fields)`：多字段 `OR LIKE %keyword%`。
- `buildOrderClause(orders)`：`[{field,direction}]`。

BaseRepository 的 `findPaginated` / `findWithKeyword` 已内置关键字分页逻辑，子类只需声明 `keywordFields` 与 `orders`。

### 5.6 BaseRepository 通用能力

[BaseRepository](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/repositories/BaseRepository.js) 提供：

- `create(data)`：动态拼 INSERT 并回查新行。
- `update(id, data)` / `patch(id, data)`：动态拼 SET，自动刷新 `updated_at`。
- `raw(sql, params)` / `rawGet` / `rawRun`：用于复杂 JOIN（如随访带患者信息）。

> 注意：`update` 会把 `data` 中除主键外的所有字段写入，Service 层负责做 `...existing` 合并以实现「部分更新」语义（见 [PatientService.update](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/services/PatientService.js#L58-L81)）。

### 5.7 时间工具

[utils/time.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/utils/time.js)：`timeToMinutes` / `minutesToTime` / `isTimeOverlap`（核心冲突判定）/ `isValidTimeRange` / `today` / `getMonthRange` 等。冲突检测依赖 `isTimeOverlap` 的「半开区间」语义——时间刚好衔接不判冲突（`s1 < e2Safe && s2 < e1Safe`）。

## 6. 数据库设计

建表与种子在 [db/index.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/db/index.js)，开启 `journal_mode=WAL` 与 `foreign_keys=ON`。

| 表 | 说明 | 关键外键 |
|----|------|----------|
| `patients` | 患者档案 | — |
| `followups` | 随访记录 | `patient_id → patients(id) ON DELETE CASCADE` |
| `schedules` | 排班/预约 | `patient_id → patients(id) ON DELETE SET NULL` |

三表均含 `created_at` / `updated_at`（`datetime('now','localtime')` 默认值），状态字段有默认值（`active`/`pending`/`scheduled`）。

**字段约定**：时间字段存字符串（`YYYY-MM-DD`、`HH:mm`），不使用时间戳，便于直接比较与展示。

## 7. 业务模块详解

### 7.1 患者管理

- 列表分页，支持 `keyword`（姓名/电话/诊断）、`status`、`department` 筛选，按 `created_at DESC`。
- `getAll` 仅返回 `active` 患者的精简字段（下拉选择用）。
- `getDepartments` 返回去重科室列表。
- 创建时 `name` 必填，其余字段缺省取默认值。

### 7.2 随访管理

- 列表通过 `LEFT JOIN patients` 一次性带出 `patient_name/gender/age/phone/diagnosis/patient_department`。
- 筛选维度：`patient_id`、`status`、`department`（按患者科室）、`doctor`（随访医生或患者医生）、`keyword`（患者姓名/随访内容）。
- `todayPending`：按当日日期 + `pending` 状态查询，TTL 60s。
- 创建时校验患者存在性，`doctor` 缺省回退到患者主治医生。
- `updateStatus` 仅改 `status` 与 `result`，是轻量补丁接口。

### 7.3 排班管理与冲突检测（核心）

冲突检测算法位于 [ScheduleRepository.findConflicts](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/repositories/ScheduleRepository.js#L81-L110)：

1. 查同日且状态为 `scheduled`/`ongoing` 的排班（可 `excludeId` 排除自身，用于更新场景）。
2. 对每条用 `isTimeOverlap` 判断时间重叠。
3. 重叠则分别检测三类冲突：`doctor`（医生撞期）、`room`（诊室占用）、`patient`（患者重复预约），可同时命中多条。
4. `checkConflict` 接口（[ScheduleService.checkConflict](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/services/ScheduleService.js#L51-L70)）返回 `{hasConflict, conflicts}`，**不抛错**；而 `create/update` 命中冲突会**抛 `conflict`（HTTP 409）**。

排班还有两个特色查询：`getDepartments`/`getDoctors` 合并 `patients` 与 `schedules` 两表去重；`listByMonth` 用 `[月初, 下月初)` 半开区间跨月正确计算。

## 8. 前端架构

### 8.1 请求层 [utils/request.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/frontend/src/utils/request.js)

基于 Axios 的增强封装，能力包括：

- **统一响应解包**：拦截器判断 `res.code !== 0` 时弹 `ElMessage.error` 并 reject，成功直接返回 `res`。
- **自动重试**：网络错误/5xx 指数退避重试（默认 2 次）。
- **防重复请求**：GET 请求按 `method+url+params+data` 生成 key，重复请求取消前者（`cancelDuplicate`）。

`request.cancelAll()` 在 [MainLayout](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/frontend/src/layouts/MainLayout.vue#L70-L72) 卸载时调用，避免内存泄漏。

## 9. 二开实战指南

### 9.1 新增一个完整业务模块（以「用药记录 medications」为例）

**后端五步：**

1. **建表**：在 [db/index.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/db/index.js) 的 `initDb` 与 [tests/helper.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/tests/helper.js) 同步追加 `CREATE TABLE medications`。

## 13. 已知问题与修复建议

### 13.1 响应/错误处理插件未生效（Fastify 封装问题）

**现象（已实测验证）：**

- 所有成功请求 HTTP 恒为 `200`；`POST` 创建接口本应返回 `201`，但 `_statusCode:201` 字段泄漏进响应体，HTTP 仍为 200。
- `AppError`（404/400/409）走的是 Fastify 默认错误处理器，响应体为 `{ statusCode, code:"−1"(string), error, message }`，**不含 `data` 字段**——导致 `conflict('预约存在冲突', {conflicts})` 的冲突明细无法返回，`schedules.test.js` 中 `data.conflicts` 断言失败。

**根因：** [server.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/server.js#L23-L39) 用 `fastify.register(responseHandler)` 注册插件，而 `responseHandler` 内部通过 `fastify.addHook(preSerialization)` 与 `fastify.setErrorHandler(...)` 添加的钩子被封装在该插件子作用域内，**不会作用于兄弟注册的 `patientRoutes/followupRoutes/scheduleRoutes`**（Fastify 的封装是父子继承、兄弟隔离）。

**修复方案（任选其一）：**

- **方案 A（推荐，改动最小）**：把插件改为 `fastify-plugin` 包装以打破封装。安装 `fastify-plugin`，将 [responseHandler.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/middleware/responseHandler.js) 导出改为 `export default fp(responseHandler, { name:'response-handler' })`。
- **方案 B**：在 `server.js` 中直接在根实例上挂钩子，而非封装进 `register`：
  ```js
  // 替换 fastify.register(responseHandler)
  fastify.addHook('preSerialization', async (req, reply, payload) => {
    if (payload && payload._statusCode) {
      reply.code(payload._statusCode);
      const { _statusCode, ...rest } = payload; return rest;
    }
    return payload;
  });
  fastify.setErrorHandler((error, req, reply) => { /* 同 responseHandler 内逻辑 */ });
  ```
- **方案 C**：将路由 `register` 放到 `responseHandler` 插件注册之后的同一上下文链中（如定义一个组合插件统一注册）。

修复后：POST 创建返回 HTTP 201 且体不含 `_statusCode`；`AppError` 返回 `{ code:-1, message, data }`，冲突明细 `data.conflicts` 正常返回，全部测试可通过。

### 13.2 其他注意点

- `dashboard` 的 `completionRate` 依赖 `patientStore.pendingCount`（当前患者列表的 pending 计数，语义偏弱），二开重写看板时建议改为基于随访总数的真实完成率统计。
- `FollowupRepository.listTodayPending` 用 `new Date().toISOString().split('T')[0]`（UTC 日），与系统其他位置的 localtime 不完全一致，跨日时可能偏差，二开可统一为 `today()`。
- 删除患者会级联删除其随访（`ON DELETE CASCADE`），排班的 `patient_id` 仅置空（`ON DELETE SET NULL`），但 `patient_name` 字段保留，二开需注意展示一致性。
