# 医疗预约随访系统 · 接口文档

> 后端服务 Base URL：`http://<host>:3001/api`（开发态前端经 Vite 代理 `/api` 转发）
> 所有接口均为 JSON。本文档基于代码与实测编写，覆盖 27 个接口。

---

## 0. 通用约定

### 0.1 Base URL 与前缀

| 模块 | 前缀 |
|------|------|
| 系统 | `/api` |
| 患者管理 | `/api/patients` |
| 随访管理 | `/api/followups` |
| 排班管理 | `/api/schedules` |

### 0.2 统一响应格式（成功）

```json
{
  "code": 0,
  "data": {},
  "message": "success"
}
```

- `code`：业务码，`0` 表示成功（见 [constants](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/constants/index.js#L45-L50)）。
- 分页接口的 `data` 结构为 `{ list, total, page, pageSize }`。

### 0.3 统一响应格式（错误 · 实际行为）

```json
{
  "statusCode": 404,
  "code": "-1",
  "error": "Not Found",
  "message": "患者不存在"
}
```

- `code` 为字符串 `"-1"`（业务码 `RESPONSE_CODE.ERROR=-1`）。
- `error` 为 HTTP 状态文本（`Bad Request` / `Not Found` / `Conflict`）。
- `message` 为中文错误描述。

> ⚠️ **重要说明（实测）**：由于 Fastify 插件封装机制，自定义错误处理器当前未作用于路由层，因此错误响应**不含 `data` 字段**。设计上 `conflict` 本应返回 `{ code, message, data:{conflicts} }`，但当前冲突明细 `data.conflicts` 不会出现在响应体中。详情与修复方案见 [二开手册 §13](./二开手册.md#13-已知问题与修复建议)。

### 0.4 HTTP 状态码（实际）

| 状态码 | 含义 | 触发场景 |
|--------|------|----------|
| 200 | 成功 | 所有成功请求（**包括 POST 创建**，设计为 201 但实际为 200） |
| 400 | 参数错误 | 必填缺失、时间范围非法等（`badRequest`） |
| 404 | 资源不存在 | 按 ID 查/改/删时记录不存在（`notFound`） |
| 409 | 冲突 | 排班创建/更新检测到时间冲突（`conflict`） |

### 0.5 业务码

| 值 | 含义 |
|----|------|
| 0 | SUCCESS |
| -1 | ERROR |
| -2 | UNAUTHORIZED（预留，未启用） |
| -3 | FORBIDDEN（预留，未启用） |

### 0.6 分页约定

- 默认 `page=1`，`pageSize=10`，最大 `pageSize=200`（见 [config](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/config/index.js#L19-L23)）。
- 参数以 query string 传递，均为字符串。
- 分页响应：`data.list` / `data.total` / `data.page` / `data.pageSize`。

### 0.7 通用请求头

```
Content-Type: application/json
```

当前系统无鉴权，无需 Token。

### 4.3 冲突检测

`POST /api/schedules/check-conflict`

**说明**：检测给定时间段是否与已有排班冲突。**此接口不抛错，恒返回 200**，通过 `data.hasConflict` 表达结果。建议前端在提交前调用预检。

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| date | string | 是 | 日期 `YYYY-MM-DD` |
| start_time | string | 是 | 开始时间 `HH:mm` |
| end_time | string | 否 | 结束时间 `HH:mm` |
| doctor | string | 否 | 医生（用于医生冲突判定） |
| room | string | 否 | 诊室（用于诊室冲突判定） |
| patient_id | int | 否 | 患者 ID（用于患者冲突判定） |
| exclude_id | int | 否 | 排除自身排班 ID（更新场景） |

**冲突类型**（[CONFLICT_TYPES](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/constants/index.js#L29-L34)）：`doctor` 医生撞期、`room` 诊室占用、`patient` 患者重复预约、`time` 时间范围非法。

**成功响应** `200`（有冲突）

```json
{
  "code": 0,
  "data": {
    "hasConflict": true,
    "conflicts": [
      {
        "type": "doctor",
        "message": "该时间段医生[李医生]已有排班：张三-复查",
        "schedule": { "id": 1, "title": "张三-复查", "...": "..." }
      },
      { "type": "room", "message": "该时间段诊室[门诊2室]已被占用：张三-复查", "schedule": {...} },
      { "type": "patient", "message": "该患者此时间段已有预约：张三-复查", "schedule": {...} }
    ]
  },
  "message": "success"
}
```

**成功响应** `200`（无冲突）：`{ code:0, data:{ hasConflict:false, conflicts:[] }, message:"success" }`

> 判定规则：仅与同日、状态为 `scheduled`/`ongoing` 的排班比较；时间采用半开区间重叠判定，**刚好衔接不判冲突**；`date` 或 `start_time` 缺失时直接返回无冲突；`end_time <= start_time` 返回 `time` 类型冲突。

### 4.7 创建排班

`POST /api/schedules`

**请求体**

| 字段 | 类型 | 必填 | 默认 | 说明 |
|------|------|------|------|------|
| title | string | 是 | — | 标题 |
| date | string | 是 | — | 日期 `YYYY-MM-DD` |
| start_time | string | 否 | "" | 开始时间 `HH:mm` |
| end_time | string | 否 | "" | 结束时间 `HH:mm`（须晚于 start_time） |
| type | string | 否 | followup | 类型 |
| patient_id | int | 否 | null | 关联患者 ID |
| patient_name | string | 否 | "" | 患者姓名（随访类型且传了 patient_id 时自动回填） |

| room | string | 否 | "" | 诊室 |

**错误响应**
- `400`：`message:"标题和日期为必填项"` 或 `message:"结束时间必须晚于开始时间"`
- `409`：`message:"预约存在冲突"`（⚠️ 设计上含 `data.conflicts`，但当前实际响应体不含 `data`，见 §0.3）

### 4.8 更新排班

`PUT /api/schedules/:id`

**请求体**：任意字段子集（部分更新）。更新到 `scheduled`/`ongoing` 状态时同样会做冲突检测，并自动用 `excludeId=当前id` 排除自身。

**成功响应** `200`：`{ code:0, data:{...}, message:"更新成功" }`

**错误响应**
- `400`：必填校验/时间范围校验
- `404`：`message:"排班不存在"`
- `409`：`message:"预约存在冲突"`

### 4.9 更新排班状态

`PATCH /api/schedules/:id/status`

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| status | string | 否 | 新状态 |

**成功响应** `200`：`{ code:0, data:{...}, message:"状态更新成功" }`

**错误响应** `404`：`message:"排班不存在"`

### 4.10 删除排班

`DELETE /api/schedules/:id`

**成功响应** `200`：`{ code:0, data:null, message:"删除成功" }`

**错误响应** `404`：`message:"排班不存在"`

### 5.3 随访类型 FOLLOWUP_TYPES

`电话随访` / `门诊随访` / `上门随访` / `微信随访` / `其他`

### 5.4 排班状态 SCHEDULE_STATUS

| 值 | 含义 |
|----|------|
| scheduled | 已排 |
| ongoing | 进行中 |
| completed | 已完成 |
| cancelled | 已取消 |

> 冲突检测仅针对 `scheduled` 与 `ongoing` 状态的排班。

### 5.5 排班类型 SCHEDULE_TYPES

| 值 | 含义 |
|----|------|
| followup | 随访 |
| clinic | 门诊 |
| meeting | 会议 |
| other | 其他 |

### 5.6 冲突类型 CONFLICT_TYPES

| 值 | 含义 | 触发条件 |
|----|------|----------|
| doctor | 医生冲突 | 同时间段同医生已有排班 |
| room | 诊室冲突 | 同时间段同诊室已被占用 |
| patient | 患者冲突 | 同患者同时间段已有预约 |
| time | 时间冲突 | `end_time <= start_time` |
