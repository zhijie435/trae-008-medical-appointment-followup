## 2. 依赖管理规范

### 2.1 版本约束策略

| 依赖类型 | 约束方式 | 说明 |
|----------|----------|------|
| 核心框架（Vue/Fastify/Vite） | `^主版本.次版本.0` | 接受补丁更新，拒绝破坏性升级 |
| 组件库/插件 | `^主版本.次版本.0` | 同上，避免 API 变动 |
| 工具类（dayjs/axios） | `^主版本.次版本.补丁` | 允许小版本更新 |
| 开发工具（Jest/nodemon） | `^主版本.次版本.0` | 稳定为主 |

> 注意：当前项目未启用 lockfile 严格模式。生产部署建议使用 `npm ci` 而非 `npm install`，确保依赖版本与 `package-lock.json` 完全一致。

### 2.2 依赖安装顺序

```bash
# 1. 安装后端依赖（better-sqlite3 需原生编译，耗时较长）
cd backend
npm install          # 或 npm ci（推荐生产）

# 2. 安装前端依赖
cd ../frontend
npm install          # 或 npm ci
```

### 2.3 原生模块说明

`better-sqlite3` 是原生 C++ 模块，安装时会触发 `prebuild-install` 下载预编译二进制：

- **网络环境好**：直接下载对应平台预编译包，无需本地编译。
- **离线/内网环境**：需提前缓存 `.node` 文件到 `~/.cache/node-pre-gyp/`，或安装 `python3` + `make` + `gcc` 本地编译。
- **跨平台部署**：在目标平台执行 `npm rebuild better-sqlite3` 重新编译。

常见平台对应文件：
- macOS x64：`better-sqlite3-v9.4.3-node-v115-darwin-x64.tar.gz`
- macOS arm64：`better-sqlite3-v9.4.3-node-v115-darwin-arm64.tar.gz`
- Linux x64：`better-sqlite3-v9.4.3-node-v115-linux-x64.tar.gz`

### 2.4 依赖审计与更新

```bash
# 安全漏洞扫描
cd backend && npm audit
cd ../frontend && npm audit

# 交互式升级（需安装 npm-check-updates）
npx npm-check-updates -i
```

## 4. 构建配置详解

### 4.1 前端构建（Vite）

配置文件：[frontend/vite.config.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/frontend/vite.config.js)

#### 4.1.1 核心配置项

```js
export default defineConfig({
  plugins: [vue()],                          // Vue SFC 编译插件
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url))  // 路径别名
    }
  },
  server: {
    port: 5173,                              // 开发端口
    proxy: {
      '/api': {                              // API 反向代理
        target: 'http://localhost:3001',
        changeOrigin: true
      }
    }
  }
});
```

#### 4.1.2 路径别名约定

| 别名 | 实际路径 | 使用场景 |
|------|----------|----------|
| `@/api/*` | `frontend/src/api/*` | 接口封装 |
| `@/composables/*` | `frontend/src/composables/*` | 组合式函数 |
| `@/stores/*` | `frontend/src/stores/*` | Pinia stores |
| `@/views/*` | `frontend/src/views/*` | 页面组件 |
| `@/utils/*` | `frontend/src/utils/*` | 工具函数 |
| `@/constants/*` | `frontend/src/constants/*` | 常量枚举 |

> 代码中**必须使用别名**而非相对路径（如 `import x from '@/utils/request'`），避免深层嵌套的 `../../`。

#### 4.1.3 开发代理说明

- 前端请求 `/api/*` → Vite 代理转发到 `http://localhost:3001/api/*`
- `changeOrigin: true`：修改请求 Host 头为 target，避免后端 CORS 校验异常
- 生产环境：由 Nginx 或网关负责 `/api` 反代，不走 Vite

#### 4.1.4 构建产物分析

```bash
cd frontend
npm run build
# 默认生成 dist/ 目录，包含：
#   index.html          # 入口 HTML
#   assets/*.js         # 打包后的 JS（含 hash）
#   assets/*.css        # 打包后的 CSS
```

如需体积分析，可临时接入：

```bash
npx vite-bundle-analyzer   # 或安装 rollup-plugin-visualizer
```

### 4.2 后端启动配置

入口文件：[backend/src/server.js](file:///Users/wuzhijie/Documents/xiaohongshu/biaozhu/trae-web-projects/008-医疗预约随访系统/backend/src/server.js)

#### 4.2.1 启动装配流程

```
1. 创建 Fastify 实例（带 logger）
2. 注册 CORS 插件（全放通，生产建议收紧）
3. 注册 responseHandler 插件（响应/错误处理）
4. 初始化 SQLite 数据库（建表 + 种子数据）
5. 创建 RepositoryFactory → ServiceFactory → ControllerFactory
6. 通过 fastify.decorate 注入各层工厂到实例
7. 注册三大业务路由（/api/patients, /api/followups, /api/schedules）
8. 挂载健康检查 /api/health 和缓存统计 /api/stats/cache
9. 监听 0.0.0.0:3001
```

#### 4.2.2 nodemon 配置（隐式）

项目未创建 `nodemon.json`，使用默认配置：
- 监听目录：`backend/` 下全部文件
- 忽略：`node_modules/`、`*.db`、`*.db-shm`、`*.db-wal`
- 延迟重启：默认 1s

如需自定义，在 `backend/` 下新增 `nodemon.json`：

```json
{
  "watch": ["src"],
  "ignore": ["src/data/*.db", "src/data/*.db-*", "tests"],
  "ext": "js,json",
  "delay": 1000
}
```

### 6.2 ESM 兼容说明

Jest 默认使用 CommonJS 沙箱，项目启用原生 ESM 需满足：

1. `package.json` 设置 `"type": "module"` ✅ 已配置
2. 启动时注入 `NODE_OPTIONS=--experimental-vm-modules` ✅ 已配置（通过 cross-env）

> ⚠️ `--experimental-vm-modules` 是 Node.js 实验性特性，在 Node 20+ 已稳定。Node 18 以下可能出现兼容性问题，建议使用 **Node.js ≥ 20 LTS**。

## 8. Git 规范与 Hooks

### 8.1 分支策略建议

| 分支名 | 用途 | 命名示例 |
|--------|------|----------|
| `main` / `master` | 生产代码，受保护 | — |
| `develop` | 集成分支，功能合并目标 | — |
| `feature/*` | 新功能开发 | `feature/medication-module` |
| `fix/*` | Bug 修复 | `fix/schedule-conflict-data` |
| `hotfix/*` | 生产紧急修复 | `hotfix/cors-origin` |
| `refactor/*` | 重构（无功能变更） | `refactor/base-repo-generic` |
| `chore/*` | 配置/工具/文档变更 | `chore/add-eslint` |

### 8.2 Commit Message 规范（Conventional Commits）

格式：`<type>(<scope>): <subject>`

| type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(patient): 新增患者导入接口` |
| `fix` | Bug 修复 | `fix(schedule): 修复更新时冲突排除自身` |
| `docs` | 文档更新 | `docs: 补充工程化配置文档` |
| `style` | 代码格式（不影响逻辑） | `style: 统一缩进为 2 空格` |
| `refactor` | 重构 | `refactor(cache): 抽象 LRU 淘汰算法` |
| `perf` | 性能优化 | `perf(list): 添加关键字查询索引` |
| `test` | 测试相关 | `test(followup): 补充分页边界用例` |

```nginx
server {
  listen 80;

    try_files $uri $uri/ /index.html;
```

### 12.2 日志配置

Fastify 使用内置 pino 日志：

| 级别 | 场景 |
|------|------|
| `fatal` | 进程即将崩溃 |
| `error` | 请求处理异常（5xx） |
| `warn` | 潜在问题（如缓存淘汰、参数警告） |
| `info` | 服务启动、请求摘要 |
| `debug` | 缓存命中/失效、SQL 语句 |
| `trace` | 最细粒度（不建议生产开启） |

生产建议：`info`；排查问题时临时改为 `debug`。

### 12.3 关键告警指标

| 指标 | 建议阈值 | 触发场景 |
|------|----------|----------|
| CPU | > 80% 持续 5 分钟 | 慢查询/内存泄漏 |
| 内存 RSS | > 512MB | 缓存无限增长 |
| 接口 P95 | > 500ms | 查询未命中索引 |
| 错误率 | > 1% | 业务异常激增 |
| 缓存命中率 | < 60% | 缓存键设计不当 |

## 15. 常见问题排查（Troubleshooting）

| 问题 | 可能原因 | 解决方法 |
|------|----------|----------|
| `npm install` better-sqlite3 失败 | 无网络 / 缺编译工具 | 切镜像 `npm config set registry https://registry.npmmirror.com`；或安装 python3+make+gcc |
| Jest 报错 `not supported` | Node 版本低 / 缺少 ESM flag | 升级到 Node 20+；确保用 `npm test` 启动（带 `--experimental-vm-modules`） |
| 前端开发跨域失败 | Vite 代理未生效 | 确认前端走 `/api/*` 路径而非 `http://localhost:3001/*` |
| 生产 SQLite 锁定错误 | 多进程写入 | 确保单实例；或 WAL 模式下增加 `busy_timeout=5000` |
| Docker 启动后数据库空 | Volume 未挂载 | 检查 `db-data` volume 是否正确映射 `/app/src/data` |
| 缓存数据不更新 | 写操作未失效缓存 | 检查 Service 层是否调用 `_invalidateCache()` |
| Vite HMR 失效 | 路径大小写 / 文件监听 | 确认文件名大小写一致；macOS 用 `fsevents`，Linux 调 `fs.inotify.max_user_watches` |
